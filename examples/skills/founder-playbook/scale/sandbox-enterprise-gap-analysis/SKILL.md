Run a per-account enterprise-procurement gap analysis through a demesne pipeline. You (the host) author one orchestrator prompt and launch a single slow-tier `sandbox_agent`; it inventories the vendor's current state once, then fans out a two-stage per-account pipeline — an open-web demand child then a mounted-evidence scoring child — and a reducer sequences the closes. The deliverable is markdown in `/out`: a sequenced gap-closure plan plus a per-account gap matrix. There is no code or branch landing — the plan is handed off to `sandbox-enterprise-procurement-pack` (docs/SLAs) and `sandbox-feature-work` (technical infrastructure).

**Watch out (cross-cutting):** `sandbox_research` demand children have open web but a FRESH private workspace and NO `/in` mounts — they cannot see the vendor's repo, so they build the *demand* only; scoring against reality is a separate mounted `sandbox_agent`. A "met" backed only by a document (not observed telemetry) is at most PARTIAL — an uptime SLA or an access-review cadence the vendor can't demonstrate from logs is documented-not-enforced, and procurement will catch it. The orchestrator copies deliverables into its own `/out` itself; a `sandbox_script` copy child strands them under `/out/child/<name>/`.

## Procedure

1. **INTAKE** (orchestrator's own process). Write `/workspace/vendor-profile.md` (what the product is, hosting/data-flow, target market → which frameworks bite: SOC 2 / ISO 27001 / GDPR / HIPAA) and `/workspace/accounts.md` — one block per account: name (or ICP archetype), why demanding, and buyer-size class (mid-market / enterprise / F500-regulated), which sets the demand bar. If the user named fewer/more than three, honour that; three is the playbook default.

2. **BASELINE** (`sandbox_agent`, medium, name `baseline`, inherits `/in` mounts). Reads the mounted `/in/<repo>`, `/in/<docs>`, and `/in/<procurement-pack>` (if the sibling skill produced one) and scores each row of the embedded baseline checklist (see "Writing the orchestrator prompt") **met / partial / missing** with an evidence path (`file:line`, URL, or a monitoring artefact) into `/workspace/current-state.md`. Messy real-world files are expected — log anything unparseable in a `## Unreadable` section rather than scoring it MET by omission. For the "dependencies with known vulnerabilities" row only, spawn a deterministic `sandbox_script` (image per ecosystem, `egress: package-managers`, e.g. `npm audit`/`pip-audit`/`govulncheck`) — never an LLM guess. BASELINE runs concurrently with step 3 (mounts vs web, no dependency).

3. **DEMAND** (`sandbox_research` fan-out, medium, one per account, name `demand-<slug>`, open web, no mounts). Embed the full vendor-profile prose and the named-account framework in each prompt — the child has no mounts. It grounds THIS buyer's actual procurement requirements in public signals and writes `/out/child/demand-<slug>/demand.md`. For a named buyer: find the buyer's own supplier/vendor-requirements page and standing program (e.g. Microsoft SSPA, its current program *version*, and whether ISO 27001 is "may be" vs "will be" required), the trust-center network it queries, insurance minimums + Additional Insured asks, VPAT/WCAG target, and self-attestation vs independent-assessor requirement — each with URL + quoted excerpt + access date. For an ICP with no named buyer: build the archetype demand for that class (e.g. "F500 regulated-healthcare buyer" → HIPAA §164.312 + Type II + ISO 27001 + $5M insurance). Dispatch `background: true`, poll `sandbox_wait` (≤8 in flight; blocking calls are issued one per turn and run sequentially).

4. **SCORE** (`sandbox_agent` fan-out, medium; slow for F500-regulated accounts; one per account, name `score-<slug>`, inherits `/in` mounts, no web). **Barrier:** dispatch `score-<slug>` only after both `baseline` and its own `demand-<slug>` reach `succeeded` — it reads the demand at `/in/previous-jobs/demand-<slug>/demand.md` and the inventory at `/workspace/current-state.md`, cross-checks against the mounted evidence, and writes `/out/child/score-<slug>/gaps.md`: every requirement from that buyer's demand graded met/partial/missing with evidence, a one-line gap statement, and a first-cut effort (S/M/L) and owner (docs / eng / legal-compliance / ops) to close. Because demand children finish at different times, dispatch each score child as its pair completes — a per-account pipeline, ≤8 in flight total.

5. **REDUCE** (`sandbox_agent`, slow, name `reduce`). **Barrier:** only after all `score-*` finish. Reads every `/in/previous-jobs/score-<slug>/gaps.md` plus `/workspace/current-state.md` and writes the deliverable into its `/out`: a per-account gap matrix, a cross-account section (gaps shared by ≥2 accounts), and the **sequenced gap-closure plan** ordered by *accounts-unblocked ÷ effort* — but with long-lead compliance items (SOC 2 Type II 9–20 mo, ISO 27001) pulled to the front and sequenced by calendar lead time, not effort, because a top-priority quick win still can't beat a hard gate that takes a year. Each work item names the sibling skill it hands off to.

6. **AUDIT** (`sandbox_agent`, slow, name `audit`, fresh context — never the reducer scoring its own work). **Barrier:** only after `reduce` is terminal; reads the reducer output at `/in/previous-jobs/reduce/` + the score gaps + `current-state.md` and flags: any MET with no evidence path, any demand requirement with no citation, and any capability claim (SLA uptime, access-review cadence, DR test date) marked MET on a document alone rather than telemetry. On hard errors (fabricated MET) the orchestrator runs at most **2** reducer fix rounds — each re-run is a distinct child (`reduce-r2`) reading the audit flags, so the shipped plan is the post-audit version, not the pre-audit draft.

7. **DELIVER** (orchestrator's own process). Assemble the declared Output contract from the child dirs yourself — do not delegate the copies to a `sandbox_script` child, which would write only to its own `/out/child/<name>/`:
   - Let `<reducer>` be the last reducer to run — `reduce-r2` if an audit fix round ran, else `reduce`. `cp /out/child/<reducer>/per-account-matrix.md /out/` and `cp /out/child/<reducer>/gap-closure-plan.md /out/`, folding the surviving audit flags into the plan's `## Audit caveats`.
   - Per account, enumerate the per-account evidence trail: `mkdir -p /out/accounts/$slug && cp /out/child/demand-$slug/demand.md /out/accounts/$slug/demand.md && cp /out/child/score-$slug/gaps.md /out/accounts/$slug/gaps.md` for every `$slug`.
   - `cp /workspace/current-state.md /out/current-state.md` (the baseline inventory).
   - Write `/out/metadata.json` (accounts, buyer classes, tiers, run date, program versions observed). Print `DONE`.

## Writing the orchestrator prompt

Brief it as a complete document. Terse prompts produce a scorer that grades documents instead of enforcement, and a research child that reports generic "SOC 2 good" instead of this buyer's actual gate.

1. **Vendor + accounts** — the mounts (`/in/<repo>`, `/in/<docs>`, `/in/<procurement-pack>`), the target market, and the three named accounts/ICPs with buyer-size class each.
2. **The embedded baseline checklist** (the durable demand floor; the demand child refreshes account-specifics on top): SIG/SIG-Lite or CAIQ/CAIQ-Lite answer library; **SOC 2 Type II** (Type I is a bridge only, "committing to Type II within 12 months"); **ISO 27001:2022** (EU/F500 price of admission; deflects 70–90% of questionnaires); annual 3rd-party **pen test** (≥40 hrs, findings + re-test evidence); **cyber/Tech E&O insurance** $1M–$5M, $5M+ for F500, customer as Additional Insured; **DPA + public subprocessor list** (dated, Article 28, general-authorization + objection right — and the list must actually match reality); **data residency** statement (SCCs / UK IDTA); **uptime SLA** 99.9% + service credits; **support SLA** P1–P4 (P1/P2 24×7, response ≠ resolution as separate commitments); **breach notification** 24–48h contractual; **BC/DR plan** + annual test with RTO/RPO; **VPAT/ACR** to WCAG 2.1 AA (public-sector / F500-adjacent); **trust-center presence** (SafeBase/Whistic/Vanta — deflects most for SMB, only 10–15% for F500).
3. **The named-account demand framework** the `demand-<slug>` child answers per buyer: (a) does the buyer publish a formal supplier program (SSPA-style) or run ad-hoc legal review? track the program *version*, not a snapshot; (b) which trust-center network does its TPRM team query, is the vendor's profile live there? (c) independent third-party assessment vs self-attestation, and which assessor credentials count; (d) VPAT + WCAG version; (e) insurance minimums + Additional Insured for that buyer's contract template (headline asks negotiate down — $20M→$5M is common); (f) standing-privilege scrutiny — JIT vs standing vendor access (Target-breach lesson); (g) subprocessor / data-flow disclosure depth (F500 reviews reach into the vendor's *own* subprocessors' compliance). **Worked calibration to embed verbatim:** Microsoft SSPA — new suppliers can't start work until the annual cycle completes; supplier picks a Data Processing Profile scoping which DPR controls apply; self-attestation within 90 days (missed → Red); higher-risk suppliers (subprocessors, SaaS, card handlers) need an independent assessor letter, with ISO 27701/27001 as accepted substitutes; and as of v12 (2026) ISO 27001 for SaaS moved from "may be" to "will be" required — the point being requirements tighten over time, so the pipeline tracks program version.
4. **Buyer-size differentiation** the demand child must apply: mid-market ($20K–$100K ACV, VP/Director sign-off, 2–5 mo, lighter VSA/CAIQ-Lite, "buys to reduce risk," Type I bridge often acceptable); enterprise (RFP + legal + security audit + multi-round, probes subcontractor access & data-flow mapping, "compliance/regs/incident response"); F500-regulated (standing programs, trust-center deflects least, $5M+ insurance, program-version tracking, subprocessor-chain review).
5. **The scoring rubric** for `score-<slug>`: MET = artefact exists AND current AND evidenced by a path/URL/telemetry — and for any capability claim (SLA, access review, DR), MET *requires* observed telemetry, else PARTIAL (documented-not-enforced); PARTIAL = exists but stale/incomplete/aspirational; MISSING = absent. Every grade carries evidence, a gap statement, effort (S/M/L), and owner.
6. **The sequencing formula** for `reduce`: priority = accounts-unblocked ÷ effort, then override with lead-time-first ordering for hard gates (Type II observation window, ISO cert) — say plainly that calendar time, not effort, sets their start. Each item names its handoff sibling.
7. **The pipeline contract** — steps 1–7, the background fan-out (≤8 in flight via `sandbox_wait`), the two barriers (score waits on baseline+its demand; reduce waits on all scores), and the fresh-auditor / 2-round fix cap.

## Output contract

```
/out/
  gap-closure-plan.md        # deliverable: sequenced closes, one item per gap, handoff sibling named
  per-account-matrix.md      # requirement × account grid, met/partial/missing + evidence
  accounts/
    <slug>/demand.md         # this buyer's demand profile (cited)
    <slug>/gaps.md           # scored gaps for this account
  current-state.md           # baseline inventory (met/partial/missing + evidence)
  metadata.json              # accounts, buyer classes, models, run date, program versions
```

`gap-closure-plan.md` sections in order: **TL;DR** (the 3–5 highest-leverage closes + any hard gate that must start now regardless of rank); **Method** (accounts, buyer classes, what "met" required, known limits); **Cross-account gaps** (shared by ≥2 accounts — highest leverage); **Sequenced plan** (ordered work items: gap, accounts unblocked, effort, owner, lead-time flag, handoff sibling); **Per-account readiness** (a go/no-go paragraph per account); **Audit caveats** (surviving auditor flags).

## Launching the orchestrator

- **`directories:` is effectively mandatory** — mount the vendor's `/in/<repo>` and `/in/<docs>`, plus the `sandbox-enterprise-procurement-pack` output as `/in/<procurement-pack>` if it exists. With nothing mounted, BASELINE scores everything MISSING and the run is worthless; state the mounts in the prompt.
- **Slow** tier for orchestrator, `reduce`, `audit`, and `score-<slug>` for F500-regulated accounts; **medium** for `baseline`, `demand-*`, and mid-market/enterprise `score-*`.
- Dispatch demand + score + baseline children with `background: true` and poll `sandbox_wait` (≤8 in flight) — blocking children issued one per turn run sequentially, defeating the fan-out.
- Child names (DNS-1123: lowercase, digits, interior hyphens, ≤40 chars): `baseline`, `demand-acme-health`, `score-acme-health`, `reduce`, `audit`.

Build a per-customer lock-in profile for a company's top ~10 accounts, reduce the profiles to a lock-in pattern analysis and a deepening plan, and emit a prioritised native-integration build backlog. You author one orchestrator prompt, launch a slow-tier `sandbox_agent`, and it runs the full map-reduce pipeline autonomously over a MOUNTED corpus of customer evidence. The deliverable is `/out/REPORT.md`, `/out/profiles.jsonl`, and `/out/backlog.json`; there is **no code landing** ‚Äî this skill designs the integrations to build, and `backlog.json` is handed to a separate `sandbox-feature-work` run that implements them.

**Watch out (cross-cutting):** Switching-cost estimates flatter the founder ‚Äî everyone wants to believe their customers are trapped, so unaudited profiles overstate lock-in; the fresh auditor stage exists to catch that and its downgrades are load-bearing, not optional. Evidence rarely arrives cleanly split by customer: if the intake manifest doesn't pin every file to exactly one account (or log it as unassignable), a map child silently conflates two customers' automations. Report-only: no host systems are touched and no branch is landed here.

## Procedure

1. **Intake & grouping (orchestrator's own process).** List every file under the mounted evidence dir. Group files by customer ‚Äî one subdirectory per customer if the founder organised it that way, else infer the account from account-note headers, export keys, or filenames. Write `/workspace/customers.jsonl` (one record per customer: `{"customer_id": "acme", "evidence_paths": ["/in/<mount>/acme/..."], "tier_note": "top-10 rank / ARR if known"}`). Log every file you cannot confidently assign to `/workspace/intake-log.md` rather than guessing ‚Äî a misassigned file corrupts one profile silently. One customer = one map unit; batch 2‚Äì3 thin-evidence customers into a single child only if each customer's evidence is well under context budget.

2. **Write the op spec at `/workspace/op.md`.** Include the per-customer operation in plain English and the **locked profile schema** every `profile.json` must conform to (schema and switching-cost method spelled out in "Writing the orchestrator prompt"). Finalise it before any map child runs: a child emitting `switching_cost` as a bare string while others emit the four-part object produces profiles the reducer cannot aggregate.

3. **MAP ‚Äî one medium-tier `sandbox_agent` per customer** (`name=profile-acme`, `profile-globex`, ‚Ä¶; lowercase DNS-1123: letters, digits, interior hyphens, ‚â§40 chars). Dispatch each with `background: true`, collect its `job_id`, poll with `sandbox_wait` (`timeout_seconds: 120`), keeping **‚â§8 in flight** ‚Äî a host-resource guard, not a demesne cap; for >8 customers launch a replacement as each finishes. Blocking calls issued one per turn run sequentially, so background dispatch is what runs the profilers concurrently. Each child's prompt embeds: its customer's exact evidence-path list, the full `/workspace/op.md`, the locked schema verbatim, and instructions to write `/out/profile.json` (schema-compliant) and `/out/log.md` (files it could not parse, fields it could not determine, and the evidence it relied on). `log.md` is required ‚Äî a child that infers deep lock-in from two support tickets must say so, or the report claims certainty it does not have.

4. **REDUCE ‚Äî one slow-tier `sandbox_agent` (`name=reducer`) only after every map job is terminal.** This barrier holds: `sandbox_wait` on all map `job_id`s before dispatching. The reducer reads siblings at `/in/previous-jobs/profile-*/profile.json` (files appear there only once each sibling completes). It concatenates them into `/workspace/profiles.jsonl` (flagging, not dropping, schema-divergent records) and drafts `/out/REPORT.md` plus `/out/backlog.json` ‚Äî the pattern analysis (which integration TYPES create the deepest lock-in for THIS product), the deepening plan, and the build backlog. The reducer only reduces; it never re-profiles a customer.

5. **PRESSURE-TEST ‚Äî one fresh slow-tier `sandbox_agent` (`name=auditor`) after the reducer's draft exists.** The auditor is a fresh context that never produced a profile: it reads `/workspace/profiles.jsonl` and the draft `REPORT.md`, and for every "deep" lock-in claim and every switching-cost estimate asks "would this customer actually struggle to leave, or are we flattering ourselves?" ‚Äî checking each against the cited evidence, downgrading claims that rest on thin or absent evidence, and writing `/out/audit.md` (per-claim: upheld / downgraded / unsupported, with reason). The reducer runs once more reading `audit.md` to fold in downgrades. **Cap the audit‚Üírevise loop at 2 rounds**; unresolved disputes are recorded in the report's caveats, not litigated further.

6. **Deliver.** In the orchestrator's own process, `cp` the final `REPORT.md`, `profiles.jsonl`, `backlog.json`, and `audit.md` into the orchestrator's own `/out`. Do not delegate this copy to a `sandbox_script` child ‚Äî its `/out` is `/out/child/<name>` and the files would strand. Also copy `customers.jsonl` and write `/out/SUMMARY.md` (customers profiled, files unassigned, schema-drift flags, claims downgraded by the auditor, map children spawned). Print `DONE`.

## Writing the orchestrator prompt

Brief it as a complete document:

1. **The corpus and the goal** ‚Äî that the mount holds top-customer evidence (usage exports, support threads, integration configs/webhook logs, account/CS notes), where it is mounted, and that the goal is a lock-in map + deepening plan + native-integration backlog, per top-10 account.
2. **The locked profile schema** ‚Äî every `profile.json` carries: `customer_id`, `source_paths`, `automations` (list of `{name, what_it_automates, built_on}` ‚Äî the product feature/API each is built on), `integrations` (list of `{external_system, direction: inbound|outbound|bidirectional, mechanism: native|api|webhook|manual_export, criticality}`), `team_workflows` (list of `{workflow, teams_involved, frequency, product_dependency}`), `switching_cost` (the four-part object below), `lock_in_depth` (`surface|moderate|deep` with a one-line justification), and `evidence_gaps` (what the mounted evidence could not establish). Over-specify; a vague schema yields vague profiles.
3. **The switching-cost method (state it verbatim)** ‚Äî estimate each customer's cost to leave as four rated components (`low|medium|high`, each with a one-line rationale tied to cited evidence): **rebuild effort** (re-creating their automations elsewhere), **data migration** (extracting/porting accumulated data), **retraining** (users/teams relearning a replacement), and **workflow disruption** (business processes that stall during a switch). The overall estimate is a reasoned roll-up, not an average ‚Äî name which component dominates.
4. **The integration-type taxonomy for the pattern analysis** ‚Äî the reducer classifies lock-in mechanisms into: native integrations, API/webhook-built automations, embedded team workflows, accumulated data/config, and manual-export dependencies ‚Äî then ranks which TYPES create the deepest lock-in FOR THIS PRODUCT (depth √ó prevalence across the group), with evidence. Per the playbook, the deepest lock-in is customers building *on top of* the product via native integrations/APIs/webhooks/SDKs.
5. **The deepening plan** ‚Äî identify customers currently using the product only at the surface (`lock_in_depth: surface`) and, for each, the specific move that would deepen them (a named integration to enable, a workflow to absorb, an automation to make native) ‚Äî not generic "engage more".
6. **The backlog item schema** ‚Äî `backlog.json` is a list of `{capability, type: integration|api|webhook|sdk, deepens_customers: [...], unlocks_new_lockin: "...", effort: S|M|L, lock_in_gain: low|medium|high}`, prioritised by lock-in gain per unit effort, ready to hand to `sandbox-feature-work`. Note in the report that this skill plans, and does not build.
7. **Map child discipline** ‚Äî embed the exact evidence-path list, full op spec, schema verbatim, "write profile.json AND log.md", and "do not silently skip files or invent fields you can't evidence ‚Äî log gaps". The reducer cannot repair structural drift.
8. **Reducer + auditor brief** ‚Äî reducer after all maps terminal; auditor is a FRESH child that never profiled; audit‚Üírevise capped at 2 rounds; downgraded claims and unresolved disputes go into the report caveats.

## Output contract

```
/out/
  REPORT.md         # lock-in audit synthesis (sections below)
  profiles.jsonl    # per-customer profiles, concatenated (anomalies annotated)
  backlog.json      # prioritised native-integration/API/webhook/SDK build backlog for sandbox-feature-work
  audit.md          # per-claim pressure-test of switching-cost and lock-in-depth claims
  customers.jsonl   # intake grouping (customer_id ‚Üí evidence_paths)
  SUMMARY.md        # run summary: profiled / unassigned files / schema flags / claims downgraded / child count
```

`REPORT.md` sections in order: **TL;DR** (3‚Äì5 bullets, written last, placed first), **Methodology** (customers profiled, evidence per customer, switching-cost method, audit rounds), **Per-customer lock-in profiles** (table: customer, depth, dominant switching-cost component, key hooks), **Lock-in pattern analysis** (which integration TYPES create the deepest lock-in for this product, ranked, cited), **Deepening opportunities** (surface customers ‚Üí the specific move that deepens each), **Build backlog** (prioritised summary, pointer to `backlog.json`), **Evidence gaps & caveats** (thin-evidence profiles, auditor downgrades, unresolved disputes).

## Launching the orchestrator

- **`directories: ["<abs path to customer evidence>"]` is mandatory.** Forget it and every map child wakes with no evidence and the pipeline profiles nothing. The founder should drop one subdirectory per customer where possible; a flat pile is acceptable but raises the unassignable-file rate ‚Äî say so.
- Map children inherit this mount and read their customer's slice of `/in/<mount>` directly. Deterministic pre-staging (e.g. unzipping export archives into `/workspace/staged/<customer>/`) is a `sandbox_script` job (`image=python`, `egress=package-managers`), never an LLM agent.
- Tier: **slow** for the orchestrator, reducer, and auditor; **medium** for the per-customer map children.
- Child names: `profile-acme`, `profile-globex`, ‚Ä¶, `reducer`, `auditor`. Lowercase letters, digits, interior hyphens only, ‚â§40 chars.

Stress-test the founder's operational centrality: enumerate every workflow, decision, and approval that routes through them, then simulate a one-week founder absence to find where the company stalls, breaks, or silently degrades. You author one orchestrator prompt, launch a single slow-tier `sandbox_agent`, and it runs the pipeline autonomously — build the bottleneck map, fan out adversarial per-cluster stress simulations, reduce to a ranked derail-risk list, fan out one fix design per derailing workflow, then re-run the absent week against those fixes with a fresh adversary. The deliverable is `/out/bottleneck-stress-test.md` plus per-workflow fix designs and the machine-readable map; there is no code landing — the fix designs feed `sandbox-founder-load-audit` automations, `sandbox-product-ops-system`, and `sandbox-feature-work`.

**Watch out (cross-cutting):** The failure this whole skill exists to catch is the *silently-degrades* case — a workflow that does not visibly stall but quietly rots (SLAs slip, a customer goes unanswered, a renewal lapses) because only the founder noticed it needed doing. A per-workflow-independent simulation misses **cascades** (approval A stalling starves workflow B mid-week); cluster the workflows by dependency and let each stress child reason across its whole cluster. And the stress child must be genuinely adversarial — a cooperative "it'd probably be fine" agent produces a reassuring, useless report.

## Procedure

1. **INVENTORY (orchestrator's own process).** If a load audit is mounted at `/in/load-audit/`, read it — its automate-entirely / human-not-founder / founder-judgment classification is the spine of the map; cross-reference every item. Also read raw founder evidence at `/in/founder-evidence/` (calendar exports, task/approval logs, org chart, escalation notes, sent-mail digests — messy real formats). Deterministic parsing of structured exports (`.ics`, CSV) may go to a `sandbox_script` (`image=python`, `egress=none`) that writes its parsed records to `/workspace/parsed-evidence.jsonl` (a named shared path the orchestrator then folds into `bottleneck-map.jsonl` — a script child's own `/out/child/<name>/` is not otherwise visible here); log any file you cannot parse to `/workspace/intake-log.md` rather than dropping it silently. Write `/workspace/bottleneck-map.jsonl` — one record per founder-routed workflow/decision/approval — with the **locked schema**: `{workflow_id, name, type: workflow|decision|approval, cadence, why_founder (habit|authority|expertise|relationship), depends_on: [workflow_id...], downstream: [workflow_id...], load_audit_bucket}`. Then group interdependent workflows into clusters and write `/workspace/clusters/<NN>.jsonl` (a list of `workflow_id`s per cluster; put isolated workflows in a catch-all cluster). Write the simulation spec to `/workspace/stress-op.md` (steps 2's rules, verbatim, for the children).

2. **STRESS-TEST — fan out one adversarial `sandbox_agent` per cluster** (`name=stress-<cluster-slug>`, e.g. `stress-billing-approvals`, `stress-cluster-03`; lowercase letters, digits, interior hyphens, ≤40 chars). Dispatch each with `background: true`, collect its `job_id`, poll `sandbox_wait` (`timeout_seconds: 120`), keep **≤8 in flight** — blocking calls are issued one per turn and run sequentially, so background dispatch is the only thing that runs the simulators concurrently. Give each child an adversarial preamble ("you are running the week from hell: the founder is unreachable, assume Murphy's law, every ambiguity resolves badly"). Each reads its cluster's workflows from `/workspace/clusters/<NN>.jsonl` + `bottleneck-map.jsonl` + `stress-op.md`, and writes `/out/stress.jsonl` — one record per workflow, **locked schema**: `{workflow_id, on_absence: stalls|breaks|silently_degrades|unaffected, cascade_from: [workflow_id...], time_to_impact, severity: 1-5, who_would_notice, derail_risk: true|false, evidence}`. `derail_risk` is true when the founder is still hands-on enough that their week away derails progress — high-severity stalls/breaks, or a silent-degradation nobody else would catch. Also `/out/week-narrative.md`: the day-by-day failure story for the cluster (this is where cascades surface).

3. **REDUCE (barrier + slow-tier `sandbox_agent`, `name=reducer`).** Spawn only after every stress job reaches a terminal state (`sandbox_wait` on all stress `job_id`s first — the `/in/previous-jobs/stress-*/` mounts register at child create but files appear only when the sibling finishes). The reducer reads every `/in/previous-jobs/stress-*/stress.jsonl`, stitches cross-cluster cascades the per-cluster children could not see, ranks all `derail_risk: true` workflows by severity × blast-radius, and writes `/workspace/derail-risk.jsonl` (the fix backlog) plus the report's map + simulation sections into `/workspace/report-parts/`.

4. **FIX DESIGN — fan out one `sandbox_agent` per derail-risk workflow** (`name=fix-<workflow-slug>`), background-dispatched ≤8 in flight exactly as step 2. Each reads its workflow's map + stress records and writes `/out/fix.md`: the failure point, then the three playbook fixes — **handoff criteria** (who takes it, the explicit trigger and decision rules that let them act without the founder), **escalation path** (what qualifies as an exception and who it routes to), **exception handling** (the messy edge cases the automation must refuse and kick to a human) — and a verdict: *tighten an existing automation* vs *build a new one* (name the target: a `sandbox-founder-load-audit` workflow-logic design, a `sandbox-product-ops-system` template, or a `sandbox-feature-work` build).

5. **FIX-VERIFY (barrier + fresh slow-tier adversary, `name=week-from-hell`).** After all fix children terminate, spawn ONE fresh adversarial child — never a fix producer scoring its own design. It reads every `/in/previous-jobs/fix-*/fix.md` and re-runs the absent week *assuming the fixes are in place*: for each, does the handoff actually fire without the founder, or is there a residual failure (an un-triggered edge case, an escalation with no owner, a silent-degradation the fix doesn't detect)? It writes `/out/verify.md` with PASS/RESIDUAL per fix. On any RESIDUAL, run one fix-revision round: re-dispatch only the failing fix children under **fresh round-suffixed names** (`fix-<workflow>-r2`, then `-r3` — reusing the step-4 name errors and poisons the sibling-mount chain), with the residual noted, then re-verify with a fresh adversary named `week-from-hell-r2` (not the earlier `week-from-hell`). The re-verifier reads the revised fixes at `/in/previous-jobs/fix-<workflow>-r2/fix.md`. **Cap at 2 rounds total**; anything still RESIDUAL after that is reported as an open risk, not hidden.

6. **DELIVER (orchestrator's own process).** `cp` the report parts, every `fix.md`, `verify.md`, `stress.jsonl`, and `week-narrative.md` into the orchestrator's own `/out/` and assemble `/out/bottleneck-stress-test.md`. Do not delegate this copy to a `sandbox_script` child — its `/out` is `/out/child/<name>/` and the files strand there. Write `/out/bottleneck-map.jsonl` and `/out/SUMMARY.md` (workflows mapped, clusters, derail-risk count, fixes verified/residual).

## Writing the orchestrator prompt

Brief it as a complete document:

1. **The inputs** — that `/in/load-audit/` (if mounted) is the load inventory to cross-reference, and `/in/founder-evidence/` holds the raw evidence; children must handle messy formats and log unparseable files, never skip silently.
2. **The bottleneck-map schema** — the locked JSONL fields above, and the rule that `why_founder` must be one of habit/authority/expertise/relationship (this drives whether a fix is delegation vs automation vs a genuine judgment call to keep).
3. **The clustering rule** — group by dependency so cascades stay inside one stress child; isolated workflows go in a catch-all cluster.
4. **The simulation spec** — the four `on_absence` outcomes, that *silently_degrades* is the highest-value catch, the adversarial "week from hell" framing, and the `derail_risk` definition (founder hands-on enough that a week away derails progress). Over-specify this; a terse spec yields a shallow "seems fine" simulation.
5. **The fix template** — handoff criteria, escalation path, exception handling, plus the tighten-existing vs build-new verdict naming the destination sibling skill.
6. **The pipeline contract** — background dispatch + `sandbox_wait` for both fan-outs (blocking children run sequentially), the reducer and fix-verify barriers, the fresh-context verifier separation, and the 2-round fix-revision cap. Every child name is **unique within the run**: each revision round re-dispatches the failing fix children and the re-verifier under fresh round-suffixed names (`fix-<workflow>-r2`, `week-from-hell-r2`, then `-r3`), because reusing a spent name errors and breaks the sibling mounts; the re-verifier reads the `-r2` fix outputs.
7. **Output contract** — the file tree below; report-only, no edits/commits/branches.

## Output contract

```
/out/
  bottleneck-stress-test.md    # the deliverable (assembled by the orchestrator)
  bottleneck-map.jsonl         # every founder-routed workflow/decision/approval
  SUMMARY.md                   # workflows mapped, clusters, derail-risk count, verify results
  stress/
    stress-<cluster>/stress.jsonl        # per-cluster simulation records
    stress-<cluster>/week-narrative.md   # per-cluster day-by-day failure story
  fixes/
    fix-<workflow>/fix.md      # one fix design per derail-risk workflow
  verify.md                    # PASS/RESIDUAL per fix from the fresh adversary
```

`bottleneck-stress-test.md` sections in order: **TL;DR** (the top derail-risk workflows and whether their fixes held, written last, placed first), **Bottleneck Map** (table: workflow × type × why-founder × load-audit bucket, cross-referenced to the mounted audit where present), **Absence Simulation** (per cluster: what stalls/breaks/silently-degrades, with the cascade chains called out — silent degradations flagged distinctly), **Derail-Risk Ranking** (the ranked fix backlog with severity × blast-radius), **Fix Designs** (per workflow: handoff criteria / escalation path / exception handling / tighten-vs-build verdict), **Residual Risks** (any fix still RESIDUAL after 2 rounds, plus workflows that are genuinely founder-judgment and should NOT be delegated), **Handoff Sequence** (the order to implement fixes, highest key-person risk first).

## Launching the orchestrator

- **`directories:`** — mount `/in/founder-evidence/` (the raw evidence dir) always; mount the `sandbox-founder-load-audit` output as `/in/load-audit/` when it exists. With neither mounted the pipeline has nothing to map and produces an empty report — if the founder has no exports, have them fill the intake questionnaire from `sandbox-founder-load-audit` first.
- **Tier**: slow for the orchestrator, reducer, and fix-verify adversary; medium for the per-cluster stress children and per-workflow fix children.
- **Child names**: `stress-<cluster-slug>`, `reducer`, `fix-<workflow-slug>`, `week-from-hell` — lowercase letters, digits, interior hyphens only, ≤40 chars, unique within the parent (never `Stress_Billing` or `fix.renewal`).

## Host-side landing

There is no code landing — the deliverable is `/out/bottleneck-stress-test.md`. Read the Derail-Risk Ranking and Residual Risks first: they name the workflows where the founder is still a single point of failure. Each fix design carries a tighten-vs-build verdict pointing at the sibling skill that implements it — route them there. The Residual Risks section is the honest core: a workflow that survives the week from hell only because the founder is judgment-critical is a keep, not a bug — say so rather than forcing a fragile handoff.

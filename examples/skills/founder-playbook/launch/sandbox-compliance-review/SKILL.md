Review a mounted codebase for the issues SOC 2 / GDPR / HIPAA audits actually probe at the code level, map each finding to a named control, and emit a sequenced remediation plan plus the controls/documentation workstream an enterprise buyer will demand. The host authors one orchestrator prompt and launches a single slow-tier `sandbox_agent`; that orchestrator selects frameworks, grounds them via one `sandbox_research` child, fans out auditors, and synthesizes. The deliverables are reports in `/out/` — **there is no code or branch landing** (fixes touching auth, secrets, or data are HUMAN-REVIEW-REQUIRED; a downstream `sandbox-feature-work` run implements them once a human signs off).

**Watch out (cross-cutting):** the `sandbox_research` child has a fresh private workspace and **no `/in` mount** — it cannot see the repo; the selected frameworks and the product's data-type inventory must go in its *prompt*, and you harvest its output from `/out/child/research-frameworks/`. Every finding must carry a real `file:line` AND a specific control ID (`SOC 2 CC6.1`, `HIPAA §164.312(b)`, `GDPR Art 32(1)(a)`) — an unmapped finding is not a compliance finding. The orchestrator copies each child deliverable into its own `/out` itself; a `sandbox_script` child strands files at `/out/child/<name>/`.

## Procedure

1. **SCOPE** (orchestrator's own process). Read the mounted repo and the founder's `target-market.md` (or the intake questions in the orchestrator prompt). Determine the data-type inventory (PII, PHI, cardholder data, EU-resident data) and select frameworks: **SOC 2** for any B2B SaaS chasing the enterprise gate; **GDPR** if EU residents' personal data is processed; **HIPAA** if PHI touches the system. Write `/workspace/scope.md`: frameworks selected, one-line rationale each, the data-type inventory, and the data-flow surfaces (APIs, logs, backups, subprocessors) that carry regulated data. Selecting a framework the product doesn't need inflates the audit; omitting one the target market requires makes the whole review worthless.

2. **GROUND** — one `sandbox_research` child (`name=research-frameworks`, medium tier, open egress). Its prompt embeds the selected frameworks + data-type inventory (it has no `/in`). It gathers current 2025–2026 expectations with URL citations: the code-level Trust Services Criteria that fail in practice (CC6 access, CC7.2 monitoring, CC8.1 change management), GDPR Art 32 + retention/deletion + subprocessor-disclosure rules, HIPAA §164.312's five technical safeguards and the addressable-vs-required distinction, and the enterprise-buyer artefact checklist (SIG-Lite/CAIQ, SOC 2 Type II vs Type I, ISO 27001:2022, third-party pen test, DPA + public subprocessor list, uptime/support SLAs, breach-notification windows, VPAT/WCAG 2.1 AA). It writes `framework-ground.md` to its `/out`; the orchestrator copies it to `/workspace/framework-ground.md`. **This is a barrier** — auditors read this file, so they cannot spawn until it lands.

3. **AUDIT** — fan out one medium-tier `sandbox_agent` per dimension, dispatched `background: true`, polled with `sandbox_wait`, **≤8 in flight** (blocking calls issue one per turn and run sequentially — background is the only way to get concurrency). Each reads the codebase, `/workspace/framework-ground.md`, and `/workspace/scope.md`, and writes `/out/AUDIT-<dim>.md` — each finding = `file:line`, severity (blocking / high / medium), a one-line exposure sketch, and the mapped control ID(s). Findings without `file:line` are rejected; anything touching authentication, secrets, or data handling is tagged **HUMAN-REVIEW-REQUIRED**. DNS-1123 names (lowercase, digits, interior hyphens, ≤40 chars):
   - `audit-access-control` — SOC 2 CC6 / HIPAA §164.312(a),(d): authn & session handling, MFA enforcement, authorization checks per request, role model, shared/generic admin or multi-user service accounts, standing vs just-in-time privilege, deprovisioning hooks. (Terminated-access lingering and shared creds are the #1 SOC 2 finding.)
   - `audit-audit-logging` — SOC 2 CC7.2 / HIPAA §164.312(b): are auth events, sensitive-record access, and admin actions logged; completeness and tamper-evidence/immutability; retention (HIPAA: 6 years); whether any review path exists or failed-login floods go unwatched. Auditors check that logs are *reviewed*, not just collected.
   - `audit-data-exposure` — GDPR Art 32 / HIPAA §164.312(c),(e): TLS 1.3 in transit, AES-256 at rest, over-broad API responses or error bodies leaking regulated fields, PII/PHI written to logs, secrets committed to code/config.
   - `audit-data-lifecycle` — GDPR retention + data-subject rights: retention periods defined and enforced by *automated* deletion (backups and archives included), erasure/access requests implementable in code, pseudonymized data still treated as personal data, and outbound calls to subprocessors (esp. AI-inference providers) that match the declared subprocessor list.
   - `audit-change-mgmt` — SOC 2 CC8.1: can a production deploy be traced to a reviewed PR approved by an authorized role (CODEOWNERS, branch protection, CI/deploy config, migration authorization). This is the sibling failure to access/logging gaps and is auditable from repo config.
   Plus **`dep-scan`** — a `sandbox_script` child (not an LLM), `image=<lang>` matching the project, `egress=package-managers`, running the ecosystem's audit tool (`npm audit --json`, `pip-audit`, `govulncheck`, `osv-scanner`) against `/workspace/repo` at pinned versions, writing `/out/dep-scan.json`. Known-vuln dependencies are what pen tests and SOC 2 flag first.

4. **SYNTHESIZE (deliverable a)** — after every auditor + `dep-scan` finishes (barrier; siblings read completed peers at `/in/previous-jobs/<name>/`), spawn a fresh slow-tier `synth-remediation` (never an auditor scoring its own work). It reads all `/in/previous-jobs/audit-*/AUDIT-*.md` + the dep scan + ground + scope, dedupes, and **drops any finding it cannot re-confirm against the cited `file:line`** (verifier-as-filter — no fix loop to cap, this is a report). It writes `/out/remediation-sequence.md`: findings ordered into **must-fix-before-enterprise-deal / next-sprint / acceptable-with-documented-justification** (the HIPAA addressable "implement-or-justify" bucket needs its justification captured, not a bare yes/no). Each item: finding, mapped control(s), severity, remediation direction, and the HUMAN-REVIEW flag where it applies.

5. **CONTROLS WORKSTREAM (deliverable b)** — concurrently with step 4, spawn a medium-tier `controls-workstream` child. It reads `framework-ground.md`, `scope.md`, the audits (for which controls are *currently failing*), and any mounted docs/evidence dir, then writes `/out/controls-workstream.md`: the documentation and controls a buyer's compliance review demands, each with current state (**present / partial / absent**, checked against the repo/docs), what it requires, rough effort, and owner. Cover the checklist grounded in research — SIG-Lite/CAIQ answer library, SOC 2 Type II roadmap (bridge from Type I within 12 months, observation window), ISO 27001:2022 where EU buyers are targeted, annual third-party pen test (≥40 hrs, with remediation evidence), cyber/Tech E&O insurance, DPA + public subprocessor page with a data-residency/transfer column, uptime SLA + credits, support SLA (P1–P4), incident-response/breach-notification policy (24–48h contractual vs GDPR's 72h vs HIPAA's 60 days), BC/DR plan + annual test, VPAT/WCAG 2.1 AA if public-sector-adjacent. This lists and sequences the workstream; it does **not draft the SLAs or support playbooks themselves** (that is `sandbox-enterprise-procurement-pack`) and does not score against named accounts (`sandbox-enterprise-gap-analysis`).

6. **COMPILE** (orchestrator's own process). Copy `scope.md`, `framework-ground.md`, `remediation-sequence.md`, and `controls-workstream.md` into `/out/` with plain `cp`, and write `/out/COMPLIANCE-REVIEW.md` tying them together (section order below), including the **Compliance in the development cycle** section: treat compliance as recurring, not a one-time project — pre-merge CI gates (dep scan, secret scan, a "no PII/PHI in logs" lint), a quarterly access review, a subprocessor-list drift check (declared list vs actual outbound calls), SLA state-transition timestamping so a breach is provable, an evidence-cadence calendar for the SOC 2 Type II observation window, and program-version tracking for buyer programs that tighten over time. The highest-leverage automation is drift detection — diffing claimed-state documents against live telemetry before a renewal or observation-period audit catches the gap. Print `DONE`.

## Writing the orchestrator prompt

Brief it as a complete document — it starts cold with only the prompt and the mounts:

1. **Goal & scope** — a code-level compliance review producing the two deliverables; audit-and-plan only, no fixes landed. Give the product's target market and, if not mounted, the intake answers: who the customers are, whether EU residents' or health data is processed, whether enterprise contracts are in play.
2. **Framework-selection rule** — SOC 2 for the B2B enterprise gate; GDPR on EU personal data; HIPAA on PHI. State which the target market forces.
3. **Pipeline contract** — the six steps, the two barriers (research → auditors; auditors+dep-scan → synth/controls), the child-naming rule, and background-dispatch fan-out (≤8 in flight via `sandbox_wait`).
4. **Audit rubric** — the five dimensions above with their control families, the `file:line`+control-ID requirement, the HUMAN-REVIEW-REQUIRED tag for auth/secrets/data findings, and the exact `dep-scan` command + `image=<lang>`.
5. **Research brief** — the frameworks + data-type inventory to embed in `research-frameworks`'s prompt (it has no `/in`), and the demand that every expectation carries a citation.
6. **Sequencing buckets** — must-fix-before-deal / next-sprint / acceptable-with-justification, and the addressable "implement-or-justify" rule for HIPAA.
7. **Controls checklist** — the buyer-artefact list in step 5, each rated present/partial/absent against the repo/docs.
8. **Boundaries & caveat** — the skip-clause siblings, and the required deliverable caveat: *this pipeline surfaces issues that commonly arise in these audits; it is not a substitute for a qualified auditor or a formal certification*.
9. **Output contract** — the tree below.

Terse prompts produce a shallow audit that finds encryption (it has a dashboard) and misses access-review, change-management, and retention gaps — the controls that actually fail SOC 2. Over-specify the rubric.

## Output contract

```
/out/
  COMPLIANCE-REVIEW.md       # executive report; section order below
  scope.md                   # frameworks selected + data-type inventory + rationale
  framework-ground.md        # cited current framework + buyer expectations (from research)
  AUDIT-<dim>.md             # per-dimension code-level findings (5 files)
  dep-scan.json              # deterministic dependency vulnerability scan
  remediation-sequence.md    # deliverable (a): sequenced, control-mapped, HUMAN-REVIEW flags
  controls-workstream.md     # deliverable (b): buyer-demanded docs/controls + present/partial/absent
```

`COMPLIANCE-REVIEW.md` section order: **1.** Scope & selected frameworks (+ data-type inventory) — **2.** Prioritized security remediation sequence (deliverable a) — **3.** Controls & documentation workstream (deliverable b) — **4.** Compliance in the development cycle (recurring checks / drift detection) — **5.** Caveats & HUMAN-REVIEW flags.

Workspace artefacts (orchestrator reads; not the deliverable): `/workspace/scope.md`, `/workspace/framework-ground.md`, `/workspace/repo/`.

## Launching the orchestrator

- **`directories: ["<abs path to repo>"]` is mandatory** — it mounts as `/in/repo` for the orchestrator and every inheriting auditor (the orchestrator stages it to `/workspace/repo` with `cp -a` so `.git`/CI config survive for the change-management audit). Without it the auditors have nothing to read.
- Optionally add the existing docs/compliance-evidence dir and `target-market.md` to `directories:`/`files:` so `controls-workstream` can rate current state; if absent it marks every artefact "absent" and says so.
- Tiers: **slow** orchestrator and `synth-remediation`; **medium** `research-frameworks`, the five auditors, and `controls-workstream`; `dep-scan` is a `sandbox_script`, no tier.
- Child names: `research-frameworks`, `audit-access-control`, `audit-audit-logging`, `audit-data-exposure`, `audit-data-lifecycle`, `audit-change-mgmt`, `dep-scan`, `synth-remediation`, `controls-workstream`.

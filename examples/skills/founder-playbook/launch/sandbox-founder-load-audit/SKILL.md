description: Inventory a founder's operational load from mounted evidence (calendar export, task lists, sent-mail digest, a week's activity log, or a filled-in intake questionnaire), then triage every recurring task, every decision that lands on the founder's desk, and every workflow that only happens because the founder personally remembers into three buckets — automate-entirely, human-but-not-founder, genuinely-founder-judgment — and emit a workflow-logic design (trigger, decision rules, output format, destination) for each automation candidate. Apply when a Launch-stage founder is the bottleneck and wants to know what to hand off — "audit my operational load", "what can I automate or delegate", "everything that only happens because I remember to do it", "triage what's on my plate", "design automations for my recurring work". Skip when the goal is the Scale-stage one-week-unavailability stress test of what breaks without the founder (use sandbox-bottleneck-stress-test, which mounts THIS skill's output), when standing up the recurring PM operating system itself (use sandbox-product-ops-system), or when triaging an incoming feedback queue (use sandbox-feedback-loop-ops).

The host authors one orchestrator prompt and launches a single slow-tier `sandbox_agent`; that orchestrator maps extractors over the founder's mounted load evidence, reduces to a deduped inventory, classifies each item into the playbook's three buckets, has a fresh auditor attack the "founder-judgment" pile, then fans out designers that spec a workflow-logic design per automation candidate. The deliverable is `/out/LOAD-AUDIT.md` plus `/out/inventory.jsonl` and `/out/designs/`; this is a report-and-design pipeline — no code, no branch, no automations are actually wired (turning a design into a running automation is a later build step for `sandbox-feature-work` or `sandbox-product-ops-system`).

**Watch out (cross-cutting):** The load-bearing failure is the founder-judgment bucket becoming a junk drawer — founders rationalise habit and control as "only I can do this", so a producer that classifies its own work will under-automate; the separate `judgment-audit` child exists to break that. The second is silent evidence loss: real exports are messy (`.ics`, exported CSVs, pasted email digests, freeform logs), so extractors must log every unparseable item rather than drop it, or the audit under-counts the load it is meant to expose.

## Procedure

1. **INTAKE (orchestrator's own process).** List `/in/load-evidence/` and write `/workspace/manifest.jsonl` (`{path, type, size_bytes}` per file). If the dir is empty or holds only the blank questionnaire, DO NOT fabricate a load — write `/out/INTAKE-QUESTIONNAIRE.md` (the questionnaire embedded below), print `NEED-INPUT: founder must fill the questionnaire, drop it in the evidence dir, and re-run`, and stop. Structured exports (`.ics` calendars, task/email CSVs) are normalised deterministically by a `sandbox_script` child (`name=normalize`, `image=python`, `egress=package-managers`) that flattens them into `/workspace/normalized/<file>.txt`; never parse `.ics`/CSV by hand in an LLM. Freeform logs and the questionnaire pass through untouched.

2. **Write `/workspace/op.md` (orchestrator).** The per-item extraction op plus the **locked JSONL schema** every `extracted.jsonl` must obey: `item_id`, `source_path`, `source_locator` (line/event/date), `title`, `kind` (`recurring-task` | `incoming-decision` | `memory-workflow`), `trigger` (what sets it off today), `cadence`, `time_cost_est`, `who_does_it_today`, `founder_knowledge_dependency` (what the founder specifically knows/holds that makes it land on them). The three `kind`s are the playbook's three load types verbatim — a recurring task, a decision that lands on the founder's desk, a workflow that only happens because the founder remembers. Tell extractors to hunt the Launch-stage tells: decisions that stack up, support requests only the founder can answer, and tasks that silently stop if the founder forgets.

3. **MAP — extractors.** Shard the manifest by context budget (~100K tokens of evidence per shard; a week's calendar+tasks is usually one or two shards) into `/workspace/shard-NN.jsonl` path-lists. Spawn one medium-tier `sandbox_agent` per shard (`name=extract-01`, `extract-02`, …). Dispatch each with `background: true`, collect `job_id`s, poll `sandbox_wait` (`timeout_seconds: 120`), **≤8 in flight** — blocking calls are issued one per turn and run sequentially, so background dispatch is the only thing that runs extractors concurrently. Each prompt embeds its exact path list, full `op.md`, and the schema verbatim; each writes `/out/extracted.jsonl` (schema-locked, one record per load item) and `/out/log.md` (files touched, items skipped, unparseable rows with reason — required; a silent skip fakes completeness).

4. **REDUCE to inventory — barrier.** Only after every `extract-NN` job is terminal (`sandbox_wait` on all of them), spawn one slow-tier `sandbox_agent` (`name=inventory`). It reads siblings at `/in/previous-jobs/extract-NN/extracted.jsonl` (files appear there only once the sibling completes) and **dedupes** the same task surfacing across calendar + email + tasks into one canonical record with merged evidence, writing `/out/inventory.jsonl`. Deduping is the whole reason this is a separate reducer, not folded into MAP: the same standup or invoice-chase shows up in three exports.

5. **CLASSIFY.** Spawn `name=classify01` (medium tier) after `inventory` completes; it reads `/in/previous-jobs/inventory/inventory.jsonl` and writes `/out/classified.jsonl` — `{item_id, bucket, confidence, reason}` — assigning each item to exactly one of `automate-entirely` (deterministic-enough that logic can run it), `human-but-not-founder` (needs a person but not the founder's specific knowledge/authority) or `founder-judgment` (genuinely needs the founder). `reason` is 1–2 auditor-readable sentences. Shard the classifier only if the inventory exceeds ~150 items.

6. **AUDIT the judgment pile — fresh child, single pass.** Spawn `name=judgment-audit` (slow tier) after `classify01`; it reads `/in/previous-jobs/classify01/classified.jsonl` + `/in/previous-jobs/inventory/inventory.jsonl` and, for every `founder-judgment` item ONLY, attacks the assignment: is this true founder judgment, or founder habit / desire for control / undocumented context that could be written down? It re-buckets what fails, appending a `challenge` note, and writes `/out/audited.jsonl` (the final buckets). This is a result-filter, not a loop — it runs once, the classifier does not get to re-defend (cap: one audit pass). A producer scoring its own buckets under-automates; keep it a fresh context.

7. **DESIGN automation candidates.** The orchestrator reads `/out/child/judgment-audit/audited.jsonl`, groups `automate-entirely` items into batches of ~5, and writes each batch to `/workspace/slices/design-NN.jsonl`. Spawn one medium-tier `sandbox_agent` per batch (`name=design-01`, `design-02`, …), background-dispatched **≤8 in flight**. Each designer reads its slice + the inventory record and, per item, produces the playbook's four-field **workflow-logic design** — **trigger** (event/schedule that fires it), **decision rules** (the branching logic, including the escalate-to-human condition), **output format** (what it emits), **destination** (where the output goes) — plus a `human_in_loop` flag and the failure/exception path. Written to `/out/design.md`. Designs are logic specs, not code.

8. **COMPILE — barrier.** After all `design-NN` jobs are terminal, spawn `name=compiler` (slow tier). It reads `/in/previous-jobs/judgment-audit/audited.jsonl`, `/in/previous-jobs/inventory/inventory.jsonl`, and every `/in/previous-jobs/design-NN/design.md`, and writes `/out/LOAD-AUDIT.md`. In the orchestrator's OWN process, `cp` the compiler's `LOAD-AUDIT.md`, the inventory, and each designer's `design.md` (into `/out/designs/`) into the orchestrator's `/out` — a `sandbox_script` child writes only to its own `/out/child/<name>` and would strand them. Write `/out/SUMMARY.md` (items inventoried, bucket counts, items re-bucketed by the audit, designs produced) and print `DONE`.

## Writing the orchestrator prompt

Brief it as a complete document:

1. **The evidence and stage.** Path (`/in/load-evidence/`), what each file is, and the Launch-stage frame: the founder is in every loop and that has become a constraint. The job is to free founder attention for decisions only a founder can make — not to remove the founder.
2. **The embedded intake questionnaire** (emitted to `/out/INTAKE-QUESTIONNAIRE.md` when evidence is thin; the founder fills it and re-runs). It must ask: *(a)* recurring tasks by cadence (daily/weekly/monthly) with a rough time-cost each; *(b)* every decision that landed on you in the last two weeks — who brought it, how often that kind recurs, what you needed to know to decide; *(c)* what only happens because you remembered — what would silently stop if you forgot for a week; *(d)* which incoming questions come to you because only you know the answer; *(e)* for each item, could someone else do it, and does it need your specific knowledge or your authority?
3. **The extraction schema** (step 2) verbatim, with the three `kind`s and the Launch-stage tells to hunt.
4. **The three buckets** defined exactly — automate-entirely / human-but-not-founder / genuinely-founder-judgment — and the audit mandate that the founder-judgment pile is challenged by a fresh child, once.
5. **The four-field workflow-logic design** — trigger, decision rules (with escalate condition), output format, destination — required for every automation candidate. Over-specify; a terse brief yields designs too vague to build from.
6. **The fan-out contract** — background dispatch, ≤8 in flight, `sandbox_wait`; the barriers before `inventory`, `judgment-audit`, and `compiler`; sibling reads at `/in/previous-jobs/<name>/`.
7. **Output contract** — the tree below; report-and-design only, no edits or commits.

## Output contract

```
/out/
  LOAD-AUDIT.md            # the audit report (section order below)
  inventory.jsonl          # canonical deduped load items, schema-locked
  SUMMARY.md               # counts: inventoried, per-bucket, re-bucketed by audit, designs
  designs/
    design-NN.md           # workflow-logic designs, ~5 automation candidates per file
  INTAKE-QUESTIONNAIRE.md  # ONLY when evidence was thin — the founder fills and re-runs
```

`LOAD-AUDIT.md` sections in order: **TL;DR** (bucket split + the top 3 automations to build first, written last, placed first), **Method** (evidence used, items inventoried/deduped/skipped), **Load inventory** (the full list, grouped by `kind`), **Triage** (the three buckets with per-item reason; a called-out subsection for items the audit moved out of founder-judgment and why), **Automation designs** (the four-field designs, sequenced by time-cost saved ÷ build effort), **Delegation candidates** (human-but-not-founder items with the handoff criteria and knowledge to transfer), **Hands off to** (note that this inventory is the input `sandbox-bottleneck-stress-test` mounts at Scale).

## Launching the orchestrator

- **`directories: ["<abs path to load evidence>"]` is mandatory.** It mounts as `/in/load-evidence/` for the orchestrator and every inheriting child; forget it and the run has no load to audit and emits only the blank questionnaire.
- Evidence is founder-private and cannot be scraped from anywhere — the founder assembles the directory (any mix of exports and the filled questionnaire). Children must handle messy real-world files and log what they cannot parse.
- Tier: **slow** for orchestrator, `inventory`, `judgment-audit`, `compiler`; **medium** for `extract-NN`, `classify01`, `design-NN`; `normalize` is a `sandbox_script`, not an agent.
- Child names (DNS-1123: lowercase letters, digits, interior hyphens, ≤40 chars): `normalize`, `extract-01`, `inventory`, `classify01`, `judgment-audit`, `design-01`, `compiler`.

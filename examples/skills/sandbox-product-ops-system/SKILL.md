Design the four artifacts of a launch-stage product-management operating system and the schedule that runs them. The host authors one orchestrator prompt and launches a single slow-tier `sandbox_agent`; that orchestrator frames context, fans out one designer per artifact, gates all four through a fresh adversarial auditor, revises, and compiles. The deliverables are markdown/JSON operating-system files in `/out` plus a structured recurring-layer schedule — **there is no code or branch landing**. The sandbox designs the system; a `## Host-side finishing` step stands up the recurring layer using the host's own schedulers and connected tools, because sandboxes cannot reach live calendars, trackers, or dashboards.

**Watch out (cross-cutting):** the whole point is removing the founder from every loop (Launch challenge: decisions stack up, tasks only happen when the founder personally remembers) — an artifact that silently assumes the founder runs the ceremony or triages the bug has failed, and the auditor must catch it. The four artifacts interlock (the spec gate feeds sprint intake; triage severity feeds sprint priority; the metrics brief pulls feedback volume and the metrics-framework metrics); designed in isolation they drift out of alignment, so the auditor checks coherence, not just per-artifact quality. Copy child outputs into `/out` from the orchestrator's own process — a designer writes to `/out/child/<name>/` and delegating the copy to a script child strands the files.

## Procedure

1. **FRAME** — orchestrator's own process. Read everything under `/in/context/` (product description, team shape, current ad-hoc process notes) and, if mounted, `/in/metrics/` (a prior `sandbox-metrics-framework` run: the chosen 3–5 metrics + false-positive signatures) and `/in/scope/` (a `sandbox-mvp-scope-guardrail` scope doc). Read `/in/context/data-sources.md` — the founder's list of which analytics / billing / support tools are *actually connected on the host*. Files will be messy; log anything unparseable to `/workspace/context.md` rather than skipping it. Write `/workspace/context.md` (product, users, team, current process) and `/workspace/data-sources.md` (named real sources, or `TODO` slots flagged where none exist). If `/in/metrics/` is absent, note it — the metrics-brief designer must then flag that it is guessing metrics and cross-reference `sandbox-metrics-framework`.

2. **DESIGN** — fan out four medium-tier `sandbox_agent` children, one per artifact, dispatched with `background: true`, `job_id`s collected, polled with `sandbox_wait` (≤8 in flight — only 4 here, so dispatch all four; blocking calls issued one per turn run sequentially). DNS-1123 names — lowercase, digits, interior hyphens, ≤40 chars: `design-sprint-cadence`, `design-spec-template`, `design-triage-tree`, `design-metrics-brief` (not `design_triage`, `SprintDesign`). Each reads `/workspace/context.md` + `/workspace/data-sources.md` and writes exactly one file to its own `/out`: `sprint-cadence.md`, `spec-template.md`, `bug-triage-tree.md`, `metrics-brief-template.md`. `design-triage-tree` additionally writes `routing-table.json` — the machine-readable form `sandbox-feedback-loop-ops` consumes. Each child's full artifact spec (below) goes in its `prompt`; terse prompts produce a generic sprint blog post instead of a runnable system.

3. **VALIDATE the routing table** — deterministic, so a `sandbox_script` child (`name=validate-tree`, `image=python`, `egress=none`), never an LLM. After `design-triage-tree` completes, it reads `/in/previous-jobs/design-triage-tree/routing-table.json` and checks: valid JSON, every decision leaf routes to exactly one class, every class is one of the `sandbox-feedback-loop-ops` taxonomy slugs (`bug`, `feature-request`, `confusion`, `churn-signal`, `praise`, `escalate-human`), and no severity level is unrouted. Writes `/workspace/tree-validation.txt` (`PASS` or the list of unrouted/invalid leaves). A failing tree goes back to `design-triage-tree` in step 5.

4. **AUDIT** — spawn one fresh slow-tier `sandbox_agent` `name=audit-os` **only after all four designers reach a terminal state** (`sandbox_wait` on every designer `job_id` first — the `/in/previous-jobs/<name>` mount exists once the sibling sandbox is created, but each file appears only when that designer completes; auditing early reads empty artifacts). It is a different context from every designer — never let a designer score its own work. It reads all four artifacts at `/in/previous-jobs/design-*/` plus `/workspace/context.md`, applies the audit rubric (below), and writes `/out/audit.md`: per-artifact `PASS` / `REVISE` with specific, quoted fixes.

5. **REVISE** — for each `REVISE` artifact (and the triage tree if `validate-tree` failed), re-dispatch its designer under a **fresh round-suffixed name** (`design-triage-tree-r2`, then `-r3` on a second round — reusing the step-2 name errors and poisons the sibling-mount chain), `background: true`, poll, with the audit's fixes and its prior output appended to the prompt. Re-run the auditor and tree check under fresh names too (`audit-os-r2`, `validate-tree-r2`, then `-r3`) over the changed artifacts; the re-run `audit-os-r2` reads the revised designer's mount at `/in/previous-jobs/design-<artifact>-r2/`, not the round-1 mount. **Cap at 2 revise rounds**; after that, ship the artifact with its open issues recorded verbatim in `/out/audit.md` rather than looping.

6. **COMPILE** — orchestrator's own process. `cp /out/child/design-sprint-cadence/sprint-cadence.md /out/`, likewise the other three artifacts and `/out/child/design-triage-tree/routing-table.json`. Write `/out/OPERATING-SYSTEM.md` — the master doc: how the four artifacts interlock, the ownership map (who/what runs each recurring action, founder explicitly named only where founder judgment is genuinely required), and a pointer to `schedule.md`. Write `/out/schedule.md` — the recurring-layer contract the host wires up (see Output contract). Print `DONE`.

## Writing the orchestrator prompt

Brief it as a complete document. Embed these four artifact specs verbatim into the matching designer's prompt, and the rubric into the auditor's:

1. **Sprint cadence (`sprint-cadence.md`).** Cycle length (justify weekly vs biweekly for this team size); the ceremonies (planning, mid-cycle check, review+retro) — for each: what it produces, its inputs (specs that passed the gate + P0–P2 items from the triage tree), and **who runs it** (an agent, a teammate, or a scheduled job — the founder is an attendee/decider, not the operator, wherever possible); the intake funnel from spec-gate and triage into the sprint; and how each ceremony is scheduled so it fires without anyone remembering.

2. **Minimum-spec template (`spec-template.md`).** A fill-in template + the **gate rule**: no implementation ticket opens until a spec passes. Required fields — title; problem + linked user evidence (≥3 interview/feedback records by default, or the count set by the scope doc's amendment criteria when mounted); the single core interaction; in-scope vs explicitly-out-of-scope; success metric + target (drawn from the metrics-framework set); kill/rollback criteria; reviewer sign-off. Include a worked example filled in for a plausible feature. The gate's pass condition must be binary.

3. **Bug-triage decision tree (`bug-triage-tree.md` + `routing-table.json`).** A readable decision tree AND a machine-readable routing table. Model it like a closed routing taxonomy: a severity axis (P0 outage/data-loss/security → immediate founder escalation; P1 broken core interaction; P2 degraded; P3 cosmetic) crossed with the `sandbox-feedback-loop-ops` type classes (`bug`, `feature-request`, `confusion`, `churn-signal`, `praise`, `escalate-human`). Every leaf names one routing outcome (auto-file to sprint backlog at priority X / hold for weekly synthesis / page the founder / quarantine to `escalate-human`). No unrouted leaf; `escalate-human` is mandatory for anything outside the tree. `routing-table.json` is the artifact `sandbox-feedback-loop-ops` executes — keep the class slugs identical to that skill's taxonomy.

4. **Weekly metrics brief (`metrics-brief-template.md`).** A template whose every metric line names its **real connected data source** (from `/workspace/data-sources.md`) and a precise definition/query — never an invented source. Pull the 3–5 metrics and the benchmark targets (activation, D7/D30) from the mounted metrics-framework output; if absent, flag each as provisional and cross-reference `sandbox-metrics-framework`. Include the false-positive signature checks (signups-without-activation, revenue-without-retention, enthusiasm-without-repeat-usage), the feedback volume/theme roll-up from `sandbox-feedback-loop-ops`, and a mandatory "so what — decision this prompts" line per section so the brief drives action, not just reporting. State cadence, recipient, and how it is auto-compiled.

5. **The audit rubric** (for `audit-os`, applied per artifact): (a) **founder-bottleneck test** — does any recurring action require the founder to personally remember, decide, or be in the loop where they needn't be? Every recurring action must name a non-founder owner or be flagged. (b) **executability** — unambiguous enough that an agent or teammate runs it identically without the author present; triage tree has no unrouted leaf; spec gate has a binary pass condition. (c) **cross-artifact coherence** — spec-gate output feeds sprint intake; triage severity feeds sprint priority; metrics brief pulls the feedback volume + metrics-framework metrics; triage classes match the feedback-loop-ops taxonomy. (d) **playbook fidelity** — real ceremonies, a genuine pre-implementation gate, real named metric sources with false-positive signatures.

6. **The pipeline contract** — steps 1–6; DNS-1123 child names, each **unique within the run** — every revise round uses fresh round-suffixed names (`-r2`, `-r3`) for the re-dispatched designer, `audit-os`, and `validate-tree`, because reusing a spent name errors and breaks the sibling mounts, and the re-run auditor reads the revised designer's `-r2` mount; designers background-dispatched and polled (≤8 in flight); the barrier before `audit-os`; the deterministic tree check via `sandbox_script`; the 2-round revise cap; the orchestrator copies deliverables itself.

7. **Report-only** — do not commit code, push a branch, or stand up the recurring layer from inside the sandbox; that is the host's job.

## Output contract

```
/out/
  OPERATING-SYSTEM.md        # master doc: how the four interlock + ownership map + pointer to schedule.md
  sprint-cadence.md          # cycle, ceremonies, owners, intake funnel, scheduling
  spec-template.md           # fill-in template + binary gate rule + worked example
  bug-triage-tree.md         # readable decision tree
  routing-table.json         # machine-readable routing table for sandbox-feedback-loop-ops
  metrics-brief-template.md  # template with named real data sources + false-positive checks
  audit.md                   # per-artifact PASS/REVISE + any shipped open issues
  schedule.md                # recurring-layer contract for host-side finishing (table below)
```

`schedule.md` is a table, one row per recurring job: `{job, cadence-or-trigger, action, produces, host-wiring}` — e.g. `sprint-planning | every 2nd Monday 09:00 | open cycle, pull gated specs + P1/P2 bugs | planning doc | calendar event + agent run`; `weekly-metrics-brief | Fri 16:00 | compile from named sources | brief in team channel | scheduler + connected analytics/billing tools`; `bug-routing | on new report | apply routing-table.json | filed/paged/quarantined | sandbox-feedback-loop-ops run`.

## Launching the orchestrator

- **`directories:` mounts.** `/in/context` is **mandatory** (product/team/process + `data-sources.md`); without it the orchestrator has nothing to frame and produces generic templates. Mount `/in/metrics` (a `sandbox-metrics-framework` output) and `/in/scope` (a `sandbox-mvp-scope-guardrail` scope doc) whenever they exist — the metrics brief and the spec gate are materially better grounded when they do, and the metrics designer degrades to flagged-provisional metrics when `/in/metrics` is missing.
- **Tier:** **slow** for the orchestrator and `audit-os` (the audit is the quality gate); **medium** for the four designers; `sandbox_script` for `validate-tree`.
- Child names: `design-sprint-cadence`, `design-spec-template`, `design-triage-tree`, `design-metrics-brief`, `audit-os`, `validate-tree`. On revise, names must be unique within the run — never reuse a spent name; suffix the round instead: `design-<artifact>-r2`, `audit-os-r2`, `validate-tree-r2` (then `-r3`).

## Host-side finishing

The sandbox designed the system; it cannot reach the founder's live calendar, tracker, dashboards, or channels. Stand up the recurring layer from the host session using `/out/schedule.md` as the contract, one row at a time:

- **Ceremony scheduling** — create the recurring calendar events for the sprint ceremonies (e.g. a connected Google Calendar MCP tool) and, where a ceremony is agent-run, register it on the host's scheduler.
- **Bug routing** — wire new bug reports to run `routing-table.json` through **`sandbox-feedback-loop-ops`** on arrival; this skill defines the tree, that skill executes the recurring intake/triage/synthesis.
- **Metrics compilation** — schedule the weekly brief to compile from the named connected sources and post to the founder's channel (e.g. a connected analytics/billing + messaging tool).

Confirm the actual tool names against what is connected in your session before calling anything — never assume a specific tool exists. If nothing is connected, do not fake it: hand `/out/schedule.md` to the founder as a documented manual cadence (the same jobs, run by hand on the stated schedule) and say so plainly. Every recurring action must land on a non-founder owner or an automation; a job that quietly routes back to "the founder remembers to do it" reintroduces the exact bottleneck this system exists to remove.

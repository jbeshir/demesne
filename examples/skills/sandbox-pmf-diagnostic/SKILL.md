Skip when 3+ cycles have already stalled and the question is what to change (use sandbox-pivot-diagnostic); when defining the metrics/benchmarks/false-positive signatures before launch (use sandbox-metrics-framework); when triaging raw bug/feature feedback (use sandbox-feedback-loop-ops); when synthesising open-ended interview transcripts (use sandbox-interview-synthesis).

Diagnose product-market fit from the founder's real data. You author one orchestrator prompt, launch a single slow-tier `sandbox_agent`, and it runs the pipeline autonomously: a deterministic quant pass computes the Sean Ellis score, two parallel passes segment the qualitative signal and classify founder effort, and a bull/bear/judge stage issues the verdict. The deliverable is `/out/verdict.md` plus a data scorecard and supporting analyses; there is no code landing. Two modes: DIAGNOSTIC (survey responses mounted) and SURVEY-KIT (none mounted — emit the survey to deploy, then stop).

**Watch out (cross-cutting):** (1) The Sean Ellis percentage must be computed by the `score-calc` **script** from normalized rows — an LLM eyeballing responses invents the number; the whole diagnostic hinges on this being reproducible arithmetic. (2) The denominator is **active users only, N/A excluded** — count churned/lapsed users or the "N/A – I no longer use it" bucket into it and you manufacture a false positive out of self-selection. (3) **No single data point confirms PMF**: a one-cycle >40% under heavy founder push is exactly the false positive this pipeline exists to catch — the `bear` child and the across-cycles framing are load-bearing, not decoration.

## Procedure

1. **INTAKE & MODE** (orchestrator's own process). List the mounted inputs. If no survey-response file is present → **SURVEY-KIT mode**: write `/out/survey-kit.md` (the four-question survey + active-user definition + sampling and send-timing guidance from *Writing the orchestrator prompt* §7) and stop after step 6's copy; the founder deploys it (see *Host-side finishing*). Otherwise → **DIAGNOSTIC mode**. Write `/workspace/inputs.md` cataloguing each mounted file and flagging any that are unreadable/unrecognised — children must log unparseable rows, never silently drop them. If no founder activity log is mounted, record `effort test skipped: no activity log mounted` in `inputs.md` and carry that flag into `/out/SUMMARY.md`, so a thin run degrades visibly rather than reporting a Sean-Ellis-only verdict as if complete.

2. **NORMALIZE** — one **medium-tier** `sandbox_agent` (`name=quant-parse`). Reads the messy real-world survey export and usage/retention export from `/in/<inputs>/`, maps their arbitrary columns to canonical fields (`respondent_id`, `sean_ellis_answer` ∈ {very/somewhat/not/na}, the three free-text follow-ups, `user_id`, `event_ts`, `signup_ts`, `acquisition_source`, cohort tags), and writes `/workspace/normalized/survey.jsonl`, `/workspace/normalized/usage.jsonl`, plus `/workspace/normalize-log.md` (columns it could not map, rows dropped and why). The LLM only interprets messy structure; it does **not** compute the score.

3. **SCORE-CALC** — one `sandbox_script` (`image=python`, `egress=none`, stdlib only). Deterministic math over the normalized JSONL, so the number is auditable and not hallucinated: define **active users** as those with ≥2 core-value events in the last 14 days; join survey↔usage; compute Sean Ellis score = `count(very disappointed) / count(active respondents excluding N/A)`; segment that ratio by cohort (signup month, plan, persona tag); flag sample adequacy (≥40 respondents directional, 40–100 tighter, <30 not reportable); compute the retention curve floor per cohort (declining-to-zero = no fit; flatten-and-hold at weeks 4–8 = a foundational cohort; **tourist check** = do all sign-up cohorts converge to a shared floor or all churn identically?); rebase M-retention to M3; compute organic/referral share of signups (pull proxy). Writes `/workspace/quant.json` (every number) + `/workspace/quant.md` (human-readable). If a required field (`signup_ts`, `event_ts`, `acquisition_source`) is missing from the normalized rows, the script skips that specific metric and records the gap in `quant.md` rather than emitting a wrong number. A script, never an agent — this is arithmetic.

4. **QUAL ∥ EFFORT** — two independent **medium-tier** `sandbox_agent` children, dispatched with `background: true` and polled with `sandbox_wait` (they run concurrently only via background dispatch; blocking calls are issued one per turn and run sequentially). Both start only after step 3 finishes (they read `quant.json`).
   - `qual` — reads `/workspace/normalized/survey.jsonl` + `quant.json`. Segments the **very-disappointed** respondents by persona to isolate the High-Expectation Customer; extracts the positioning language users use for "who benefits" and "main benefit" (their words, for messaging), and mines the objections of the **somewhat-disappointed** fence-sitters. Writes `/workspace/qual.md`. If responses > ~150 across many cohorts, fan out one map child per cohort (`qual-<cohort>`, ≤8 in flight) and reduce; otherwise a single agent is correct — do not fan out small corpora.
   - `effort` — reads the founder activity log + `quant.json`. Classifies every retention-driving founder action as **PUSH** (outreach, incentives, personal follow-up, paid acquisition, promotions, heroic energy) vs evidence of **PULL** (organic/referral/word-of-mouth signups, unprompted return, deepening feature adoption). Reports the push:pull balance this cycle and, if prior-cycle scorecards are mounted (`/in/<cycles>/`), the push→pull **trend** across cycles — the shift, not the level, is the signal. Writes `/workspace/effort.md`.

5. **VERDICT (bull / bear / judge)** — after both step-4 jobs reach a terminal state (barrier: `sandbox_wait` on both `job_id`s; their `/workspace` analyses are complete only once the jobs finish). Dispatch two **medium-tier** children with `background: true`, then one **slow-tier** judge after both finish:
   - `bull` — argues the evidence is a genuine PMF pattern. `bear` — argues false positive: self-selection, problem/solution fit mistaken for PMF, score-vs-behaviour divergence (high survey % but flat retention — the Buffer 78% trap), founder-push masking as pull, calendar- vs usage-triggered survey sends, re-survey contamination. Both read the four upstream analyses from the shared `/workspace` — `/workspace/quant.json` + `/workspace/quant.md` (score-calc), `/workspace/qual.md` (qual), `/workspace/effort.md` (effort) — and each writes `/out/case.md` (harvested as `/in/previous-jobs/bull/case.md`, `/in/previous-jobs/bear/case.md`).
   - `judge` (`name=judge`, fresh context — never the producer of the analyses) reads both cases (`/in/previous-jobs/bull/case.md`, `/in/previous-jobs/bear/case.md`) + all four analyses from `/workspace` (`quant.json`, `quant.md`, `qual.md`, `effort.md`) and writes `/out/verdict.md`: one of **PMF-pattern / not-yet / false-positive-warning**, explicitly stating that no single data point confirms PMF and placing this cycle within the multi-cycle pattern; it must cross-check the false-positive signatures from `sandbox-metrics-framework` (signups-without-activation, revenue-without-retention, enthusiasm-without-repeat-usage) and preserve the bear's surviving objections as Dissent, not merge them away. No fix loop; this is a one-shot judged synthesis.

6. **DELIVER** (orchestrator's own process — not a `sandbox_script` child, whose `/out` is `/out/child/<name>/` and would strand the files). `cp` `judge`'s `verdict.md`, `bull`/`bear` `case.md`, `quant.json`/`quant.md`, `qual.md`, `effort.md`, and the normalize log into `/out/` per the contract below. Write `/out/pmf-scorecard.json` (headline numbers for the next cycle to trend against) and `/out/SUMMARY.md` (mode, sample size + adequacy, verdict, children spawned). Print `DONE`.

## Writing the orchestrator prompt

Brief it as a complete document; terse prompts produce a shallow verdict:

1. **The product and the inputs** — what the product is, its core-value action (defines "active"), where each file is mounted, and any known cohort tags. Name the founder activity log explicitly — the effort test is half the diagnostic and is silently skipped if the log is not pointed at.
2. **Sean Ellis, exact** — question: *"How would you feel if you could no longer use [product]?"* Options: Very disappointed / Somewhat disappointed / Not disappointed / N/A – I no longer use it. Score = % "very disappointed" among **active users**, **N/A excluded from the denominator**. Threshold **≥40%** "very disappointed" = meaningful signal (Ellis's ~100-startup heuristic — a diagnostic, not proof of fit). Active = used the product **≥2× in the last 14 days**. Sample: ~40 directional, 40–100 for confidence, <30 not reportable.
3. **The Superhuman follow-ups** (bundle all four): (1) the Sean Ellis question; (2) "What type of people would most benefit from [product]?" — captures positioning language in users' own words; (3) "What is the main benefit you receive from [product]?" — value language + segmentation; (4) "How can we improve [product] for you?" — surfaces fence-sitter objections. Segment the very-disappointed by persona to find the High-Expectation Customer; a roadmap split ~50/50 reinforce-core-benefit / remove-objections is the standard use of this segmentation.
4. **The effort (push-vs-pull) test** — the PUSH vs PULL taxonomy in step 4; "feeling the market pull the product out of your hands" is the target state; the push→pull **shift across cycles** is the clearest signal. Warn that non-organic growth hacks (paid, promos, referral bribes) mimic pull — call them out, don't count them as it.
5. **Corroboration & false-positive audit** — retention-curve floor level (higher = better fit), the tourist check (all cohorts churning identically = no fit), M3-rebased retention, organic/referral share; and the three `sandbox-metrics-framework` false-positive signatures the judge must test against.
6. **The verdict rubric** — PMF-pattern / not-yet / false-positive-warning; require the explicit "no single data point; pattern across cycles" statement and, when only one cycle is mounted, an honest "cannot confirm a pattern from one reading" caveat plus what the next cycle must show.
7. **SURVEY-KIT mode content** — when no responses are mounted: emit the four questions above, the active-user definition, the ~40–100 sampling target, and send guidance (**usage-triggered not calendar-triggered**; survey each user at most once to avoid re-survey contamination; reach active users, and note that the instrument structurally under-samples churned users so the score reads fit among the retained).
8. **The pipeline contract** — steps 1–6; deterministic math is the `score-calc` script (not an agent), qual∥effort use background dispatch + `sandbox_wait` with the step-5 barrier before verdict, the judge is a fresh context, and the orchestrator copies deliverables into its own `/out` itself.

## Output contract

```
/out/
  verdict.md            # DIAGNOSTIC deliverable
  pmf-scorecard.json    # headline numbers, for the next cycle to trend against
  survey-kit.md         # SURVEY-KIT mode only (the four-question survey + send guidance)
  SUMMARY.md            # mode, sample size + adequacy, verdict, children spawned
  supporting/
    quant.json          # score-calc numbers (auditable)
    quant.md
    qual.md             # HXC segmentation + positioning language + objections
    effort.md           # push-vs-pull classification + cross-cycle trend
    bull-case.md
    bear-case.md
    normalize-log.md    # unmapped columns / dropped rows
```

`verdict.md` sections in order: **Verdict** (PMF-pattern / not-yet / false-positive-warning; one paragraph; written last, placed first). **Sean Ellis Scorecard** (score, denominator method, sample size + adequacy flag, per-cohort segmentation table, the High-Expectation Customer). **Effort Test** (push-vs-pull table, this-cycle balance, cross-cycle push→pull trend). **Corroborating Signals** (retention floor + tourist check, M3-rebased retention, organic/referral share). **False-Positive Audit** (each metrics-framework signature + self-selection + score-vs-behaviour divergence, checked and rated). **This Cycle in the Pattern** ("no single data point"; where this reading sits across cycles; what the next cycle must show). **Dissent** (the bear's surviving objections, each with a concrete verification action before over-trusting the verdict).

## Launching the orchestrator

- **`directories:` is mandatory** — mount the survey responses, usage/retention export, and founder activity log (and any prior-cycle `pmf-scorecard.json`s under a cycles dir). Forget the mount and DIAGNOSTIC mode has nothing to read; forget the activity log specifically and you lose the entire effort test with no error.
- Founder-private data (survey rows, retention exports, activity logs) is mounted, never scraped — formats are flexible; `quant-parse` maps them and logs what it cannot.
- Tiers: **slow** for the orchestrator and `judge`; **medium** for `quant-parse`, `qual`, `effort`, `bull`, `bear`; `score-calc` is a deterministic `sandbox_script` (no tier). Try the single-agent `qual` before fanning out per cohort.
- Child names (DNS-1123: lowercase letters, digits, interior hyphens, ≤40 chars): `quant-parse`, `score-calc`, `qual` (or `qual-<cohort>`), `effort`, `bull`, `bear`, `judge` — never `Quant_Parse` or `score.calc`.

## Host-side finishing

Only the SURVEY-KIT path leaves the sandbox: the pipeline produces `/out/survey-kit.md`, it does not send anything. The host agent deploys it via whatever it actually has connected — e.g. a survey tool (Typeform/Google Forms) or a Gmail/email MCP tool to send the link; confirm the real tool names against what is connected in your session, and if nothing is connected, hand `survey-kit.md` to the founder to load into their own survey tool. Send **usage-triggered, once per user**. DIAGNOSTIC mode is report-only — read `verdict.md`, act on the Dissent verification actions, and re-run next cycle so `pmf-scorecard.json` builds the multi-cycle pattern the verdict depends on.
